### Key findings

- **`inference_steps=1` (FM single-step Euler) is the dominant smoothness killer.**
  - 1-step rollout on 600k v-pred ckpt: all joints diverge to ±200° within ~25 frames — robot explodes.
  - 10-step rollout on same ckpt: joints stay in ±0.6 rad range; walk shows mild HF jitter on arms; punch's left arm hits 4 rad (230°, over joint limit) but other DoFs stable.
  - **0/8 → "doesn't explode" is achieved by inference change alone.** Re-training with smooth_v1 recipe is for the remaining gap (jitter, occasional limb runaway).

- **VA's smoothness is NOT just "DDPM > FM"** — it's a stack of design choices:
  1. 50-step DDIM (we had 1-step Euler) ← biggest single factor
  2. F=16 @ 20fps = 0.8s primitives (we have F=8 @ 30fps = 0.27s) ← 3× more seams to drift across
  3. obs_mask inpainting + boundary loss ← we have neither
  4. 35-dim with no redundant velocity channel (we have 69-dim with dof_velocity that must self-consistency-match dof_angle)
  5. x0-pred + uniform t + clamp(-5,5) (we had v-pred + logit_normal + no clamp)
  6. Loss = MSE + 0.1·boundary only (we had 7-term loss soup with possibly conflicting gradients)

- **VA_motion_generation `shared_data/` is NOT in our repo** — it's the friend's private NPZ cache. We can't directly "use VA's dataset". Closest approximation: use our own `data/processed/{bones_npz, amass_babel_npz}/` (which already store original `dof_pos / root_pos / root_quat`) + borrow VA's `extract_features_v2()` to compute 35-dim on-the-fly. This is route B (deferred until we see route A's results).

- **Decision to start with route A (69-dim + smooth_v1 recipe)**, not route B (35-dim feature swap):
  - Route A is 0 lines of new code (config flips + one boundary loss already added).
  - Route A trains in 1.5h; route B is 1-2 days of dataset/trainer work.
  - If A reaches "smooth enough", B becomes ablation. If A still jitters, B is the next attack.
