## [10:00] Denoiser v3 Evaluation & Text Conditioning Diagnosis
**Summary:** Evaluated denoiser v3 (weighted sampling), found it worse than v2. Diagnosed root cause: CLIP low discriminability + guidance too low.

### What was done
- Generated comparison videos for v3 (walk/wave/kick/squat/stand)
- Signal tracing: CLIP embeddings → denoiser → CFG output
- Found raw denoiser text difference = ~4% of latent norm
- Guidance sweep: 2.5/5/10/15/30 — linear amplification
- Full DART vs G1-DART gap analysis → `logs/2026-04-08_dart-vs-g1-analysis.md`
- Diagnosis writeup → `logs/2026-04-08_diagnosis.md`

### Problems & Solutions
- **Problem [10:00]:** Denoiser v3 (with weighted sampling) performs worse than v2
  - **Root cause:** v3 used raw text inverse-frequency (2882 unique texts), many singletons getting 31x weight → effective dataset only 24%
  - **Solution:** Switched to act_cat grouping (184 BABEL categories) + sqrt-inverse frequency → effective dataset 55.3%

- **Problem [11:00]:** All text prompts generate nearly identical motion
  - **Root cause:** CLIP ViT-B/32 cosine similarity 0.73-0.89 between motion prompts. Raw denoiser output difference only ~4%. guidance=5 too weak.
  - **Solution:** Real fix requires num_primitive=4 + batch_size=128 (matching original DART), not just guidance tuning

### Key findings
- CLIP discriminability is inherently low for motion prompts — long-term may need motion-specific encoder
- Original DART uses num_primitive=4 (consecutive primitives from same sequence) — critical for text conditioning
- Original DART uses batch_size=128 — our batch_size=4096 caused 18,479 epochs (massive overfitting)
- act_cat weighting (184 categories) much better than raw text weighting (2882 texts)

## [14:30] Pipeline Verification & Canonicalization Bug Fixes
**Summary:** Systematic pipeline verification (Steps 1-5). Found and fixed critical bugs in canonicalization and normalization.

### What was done
- Created pipeline verification plan → `logs/2026-04-08_pipeline_verification_plan.md`
- Step 1 (original data): ✅ verified, found 4 bad clips → deleted
- Step 2 (DOF roundtrip): ✅ error = 0
- Step 3 (canonical roundtrip): ✅ fixed z-offset bug, now error < 0.001mm
- Step 4 (sliced primitives): ✅ fixed missing global_orient_start_6d
- Step 5 (normalization): ✅ fixed std clamping for 1-DOF joints

### Problems & Solutions
- **Problem [14:30]:** Robot sinks into ground after canonicalization
  - **Root cause:** `get_new_coordinate_g1` shifted z by pelvis height (-0.1027m), then rendering tried to compensate with `G1_CANON_Z_OFFSET`
  - **Solution:** Changed canonicalization to only shift xy (z=0): `new_transl[:, :, 2] = 0`. Set `G1_CANON_Z_OFFSET = 0.0`. Removed z-offset from `set_mujoco_from_features`.

- **Problem [15:00]:** Sliced primitives have wrong direction/orientation
  - **Root cause:** `global_orient_start_6d` (initial absolute orientation per primitive) not stored. Un-canonicalization assumed identity orientation.
  - **Solution:** Added `global_orient_start_6d` field to `process_motion_primitive_g1.py` output.

- **Problem [15:30]:** Normalization produces extreme values [-54, 19]
  - **Root cause:** G1's 1-DOF hinge joints have constant 6D rotation components → std ≈ 0 → division by near-zero
  - **Solution:** Clamped std to min=0.01 in `dataset_g1.py`. 204 features affected. SMPL doesn't have this issue (all 3-DOF ball joints).

- **Problem [16:00]:** 4 bad clips in GMR_filtered data
  - **Root cause:** GMR retarget failures — robot falls down or has wrong poses
  - **Solution:** Generated verification videos, user identified 4 bad clips, deleted them from GMR_filtered/

### Key findings
- orient_delta accumulation must use LEFT multiply (`delta @ R`, error 0.0000008) not RIGHT (`R @ delta`, error 0.046)
- Canonical coordinate system confirmed: x=lateral (hip), y=forward, z=up
- Steps 1-5 all verified ✅ — data pipeline is now correct
