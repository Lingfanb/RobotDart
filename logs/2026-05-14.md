> | 维度 | A | **B (选)** |

> | D 维度未标场景 | 阻塞 | ✅ 可延后 |
>
> **Paper framing**: "Building on BeyondMimic's classifier-guided diffusion policy and Composable Diffusion's score-addition principle, we contribute three independent affective classifiers (V/A/D). Crucially, no retraining of base diffusion is required when adding a new dimension."
>
> **Sampler 伪代码** (composable score addition):
> ```
> eps_base = diffusion(x_t, t, state)
> eps_guided = eps_base - λ_V·∇V_classifier - λ_A·∇A_classifier - λ_D·∇D_classifier - λ_wp·∇waypoint_cost
> ```
>
> **决定 / Next**: A 路线已 rule out。Scope A 验证 baseline diffusion + waypoint guidance 后接 VAD classifier head。

<a id="d14-arch"></a>

<details open>
<summary><h3 style="display:inline">🔧 架构修正 — diffusion 直接输出 action, RL tracker 退场</h3></summary>

> **背景**: 我之前画的图把 RL tracker 留在 inference 路径里,diffusion 输出 reference motion。**这个图是错的**。user 提问"diffusion 直接输出 action 来控制运动"时点出来。
>
> **正确的 BeyondMimic 3-stage 设计**:
> ```
> Stage 1 (我们 5/13-5/14 在做): RL Tracker - 用 reference 学 imitation
> Stage 2 (Scope A Step 2): Distillation - 用 tracker rollout 训 diffusion
>   - Tracker 跑大量 rollouts → 收集 (state, action) 数据
>   - BC 监督学习 diffusion: state → action
>   - Diffusion 内化 motion knowledge
> Stage 3 (paper 最终): Guided inference - state + guidance → action
>   - Diffusion (uncond) + classifier guidance
>   - 不需要 reference,只需 state
> ```
>
> **关键 invariant**: tracker 在 Stage 2 蒸馏完就退场,**不在 inference 路径里**。Inference 时只有 1 个网络(diffusion)。
>
> **决定 / Next**: 修正后的 stack 让 Scope A 的 "verify pipeline" 任务变明确:验证 Stage 1 + Stage 2 + Stage 3 串得起来。

> **背景**: 装 MDM 依赖 (easydict, diffusers, blobfile, joblib),然后用 beyond_mimic.yaml 跑 → 立刻 NaN loss。
>
> **诊断**:
> 1. 数据没 NaN/Inf,值域 [-537, 604](root_acceleration 列 346-348 有 outliers,但不在 state_data_keys 里,不影响)
> 2. `model.condition_mechanism=film` 默认值要 cond input,但 `cond_mode=[no_cond]` → `FilmTransformerEncoder.forward()` 缺 cond 参数报错。改 `condition_mechanism=add` 用 standard encoder 后通过

>
> **结果** (MDM v2 稳定收敛):
> | Step | Loss |
> |---|---|
> | 0 | 2.28 |
> | 500 | 1.83 |
> | 1000 | 0.86 |
>
> 14 steps/sec @ batch 64,GPU 0 86% util。预计 50k step ckpt 在 ~1 小时后。
>
> **分析**: Milo 自己用的应该是 `emphasis_proj=True + 大数据`,在我们 ~3000 trajs 上不够 stable。要么收集更多 rollouts (5k → 50k),要么继续用保守设置。

<a id="d14-seed-anchor"></a>

<details open>
<summary><h3 style="display:inline">🔬 Per-action seed anchoring(target_mode=relative)</h3></summary>

> **背景**: 之前 target=+0.5 是 absolute,但每个 action 的"natural neutral"在 VAD 空间位置不同(clap≈(-0.04,+0.10,+0.01), handshake≈(-0.26,-0.24,-0.13), wave_hand≈(-0.11,-0.35,-0.15))。需要 per-action 0 锚点。

**实现**: `target_mode='relative'`(默认),内部 `target_v_abs = V_seed + target_v`,以 seed 作为 0。

**验证**: clap (ΔV=0, ΔA=0) target → V_final=+0.244, A_final=+0.099,**几乎完全等于 V_seed=+0.251, A_seed=+0.102**(gap < 0.01)。锚定准确。

**Torch vs numpy V_seed 差 0.29 已 documented**:
- numpy: V_seed = -0.04(clap zero_anchor.info.yaml 报告)
- torch: V_seed = +0.25
- 根因:`motion_amplitude_ee` 在 numpy 用 median 过滤 transient,torch 用 mean(可微 proxy)→ V1 raw 偏高
- 不影响优化(torch frame 内自洽),只在 cross-check 时要注意

**决定 / Next**: paper writeup 必须明确说"V=0 = action 的 zero_anchor 处",不是物理 (0,0,0)。

<a id="d14-framework-v2"></a>

<details open>
<summary><h3 style="display:inline">🚀 Framework v2 — batched 9-grid + adaptive_close + jerk dead-end</h3></summary>

> **背景**: 5/14 v1.5 framework 跑通,但 sequential 9-grid 要 170s。开始 framework optimization 用 clap 当 test bed,目标提升 quality + speed。

**关键决策(实验驱动)**:

| 实验 | 改动 | V range | jerk p95 | 结论 |
|---|---|---|---|---|
| baseline | jerk=0, no adaptive | 0.871 | 31000 | reference |
| iter2 | jerk=1e-5 + adaptive + V-inversion bug | 0.430 | 4818 | V 崩了一半 ⚠️ |
| iter3a | adaptive_close only | 0.869 | 31156 | adaptive 完全 safe ✓ |
| iter3b | jerk=1e-7 (no inversion) | 0.633 | 18825 | V 还是掉 27% |
| iter3c | jerk=5e-8 | 0.656 | 13782 | 还是掉 25% |
| iter4 | smooth=100, jerk=0 | 0.602 | 15279 | 高 smooth 也 cap V |

**结论**(锁定的 finding):**任何 jerk-on-absolute-trajectory 惩罚都会 cap wrist 振幅 → cap V 覆盖**。v1.5 V 跟运动幅度耦合,压 jerk = 压 V。要分开必须重新设计成 delta-relative jerk(只惩罚 optimizer 引入的高频),留作下一轮。

**代码改动 lock**:
- 删 v1.3 残留 V-adaptive jerk scaling(方向反了,高 V 反而 jerk 强)
- 新增 `adaptive_close=True`(默认):`λ_close × 1/max(|ΔV|+|ΔA|, 0.1)`,小 target → close 强 10×。‖Δ‖ p95 24 → 8.2(小 target 不再 over-perturb)
- 新增 `optimize_va_batched()`:9 个 target 一次 optimizer call,batched FK(B*T 一次性 kernel),共享 autograd graph
- CLI 加 `--batched` flag

**Speed 实测**: clap 9-grid sequential 170s → batched **30.7s = 5.5× speedup**(超过我估的 2×)。8/9 action V/A range 与 sequential 一致到 0.001。

**Flag threshold 修正** (`scripts/aug_metrics_report.py`):
- collision 阈值 1e-9 → 1e-6(过滤 autograd 数值噪声 1e-9 ~ 1e-7;真碰撞 > 1e-4)
- 加 `flag_actuator`:vel_viol > 1 或 acc_viol > 1 自动 flag(real physics)
- salute 6/9 → 0/9, wave_hand 5/9 → 0/9(都是数值噪声),beckon 4/9 → 9/9(seed 真超 G1 acc 限位)

**决定 / Next**: framework v2 lock。下一轮候选: delta-relative jerk(解 V vs jerk trade-off)、full_body preset(解 V upper bound 0.65)。

<summary><h3 style="display:inline">⚠️ Picker bug: numpy/torch regressor 0.94 A offset</h3></summary>

> **背景**: 9-action 出表后发现 beckon 的 acc=457 across all 9 outputs。挖下去发现是 picker(numpy v1.5)和 optimizer(torch v1.5)对同一 NPZ 的 V/A 读数不一致。

**具体 mismatch** (beckon zero_anchor: BMLrub__rub025__0014_sitting1):
- numpy picker: V=-0.226, A=-0.035, D=-0.205, distance=0.31 from origin → 选为 zero anchor ✓
- torch optimizer: V=+0.415, A=**+0.900**, D=-0.321 — A 完全反了,而且超 0.9
- 同一 NPZ 同一 indicator,差距 0.94 A + 0.65 V

**两个 root cause**:
1. **median (numpy) vs mean (torch)** — `motion_amplitude_ee` 已知;但 A=energy_per_frame 应该一致(`(dq²).sum().mean()` 在两边都一样)。0.94 A drift **大于已知 0.29 V drift**,说明还有别的 bug 没查
2. **Picker 没过 actuator filter** — 选了 acc_viol=457 的 clip,optimizer 也修不动(legs anchored)

**为什么影响 beckon 不影响其他 action**: beckon 的 numpy A = -0.035(看起来 zero),但 torch A = 0.9(高速能量)。其他 action 两边 A 一致(差 < 0.2),所以 picker 选的 clip 也是 torch 视角的 zero anchor。

**决定 / Next**(deferred 到下次):
- repick beckon 用 torch regressor scoring + actuator filter
- 找到 numpy/torch A drift root cause(可能是 dof_pos vs features_69 dq channel 差异)
