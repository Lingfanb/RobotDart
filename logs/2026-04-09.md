# 2026-04-09 Work Log

## [12:30] DDP support for denoiser training

**Summary:** Added DDP path to `mld/train_g1_mld.py` so denoiser training can split a global batch across two GPUs via `torchrun --nproc_per_node=2`. Trainer is now rank-aware while keeping the single-GPU code path intact.

### What was done
- Added `setup_ddp()` / `cleanup_ddp()` helpers reading `RANK`/`WORLD_SIZE`/`LOCAL_RANK` from env. `init_process_group` passes `device_id` so NCCL stops guessing the device. Falls back to `(0, 1, 0)` for non-distributed runs.
- `G1MLDTrainer.__init__` accepts `rank, world_size, local_rank`. Per-rank seed = `args.seed + rank` so the two ranks sample independent batches.
- `train_args.batch_size` is treated as a **global** batch — per-rank = `batch_size // world_size`, asserted divisible.
- Dataset init uses rank-0-first + barrier so only rank 0 writes the `mean_std.pkl` cache (no race).
- VAE / checkpoint loads pass `map_location=device` so each rank loads onto its own GPU.
- EMA copy happens **before** DDP wrap (otherwise EMA's parameter names would carry the `module.` prefix). EMA update iterates over `denoiser_model_module.parameters()`, not the wrapper.
- `common_step()` takes an optional `denoiser_model` arg so validation can call the unwrapped module — only rank 0 runs validation, calling the DDP wrapper alone would deadlock.
- Stage-2+ rollout `p_sample_loop` always uses the unwrapped module since it runs under `no_grad`.
- `validate()`: rank 0 runs the loop, all other ranks call `dist.barrier()` and return. Final barrier syncs everyone before the next training step.
- `save()` / wandb / SummaryWriter / tqdm / config dump all guarded by `is_main`.
- Created [run_denoiser_single_gpu.sh](../run_denoiser_single_gpu.sh) so the single-GPU launch path is also scripted.

### Problems & Solutions
- **Problem [12:44]:** `ModuleNotFoundError: No module named 'tyro'` when running `torchrun`.
  - **Solution:** Was running from `(base)` env. `conda activate DART` first.
- **Problem [12:45]:** `FileNotFoundError: 'mvae/g1_vae_v1/args.yaml'` from VAE checkpoint load.
  - **Solution:** Path typo — actual VAE dir is `./mvae/g1_mvae/`, not `g1_vae_v1`.
- **Problem [12:46]:** DDP `_distributed_broadcast_coalesced` fails with `RuntimeError: unsupported operation: some elements of the input tensor and the written-to tensor refer to a single memory location`.
  - **Solution:** `model/mld_denoiser.py` registers the same `PositionalEncoding` instance under both `self.sequence_pos_encoder` and `self.embed_timestep.sequence_pos_encoder`, so the `pe` buffer is registered twice in the model with aliased storage. DDP can't broadcast aliased buffers. Fix: pass `broadcast_buffers=False` when wrapping. The buffer is deterministic sin/cos, identical on every rank, so syncing it is unnecessary.
- **Problem [12:45]:** `barrier(): using the device under current context` warning from NCCL.
  - **Solution:** Set `device_id=torch.device(f'cuda:{local_rank}')` in `init_process_group` after `torch.cuda.set_device(local_rank)`.

## [13:30] Dataset prefetch optimization (root cause of slow DDP)

**Summary:** First DDP run hit only ~3 it/s with very asymmetric GPU utilization (cuda:0 73%, cuda:1 26%). Diagnosed as CPU-bound `get_batch()`, not GPU heterogeneity. Rewrote `G1PrimitiveSequenceDataset` to precompute everything at init so `get_batch` becomes a couple of GPU `index_select` calls instead of a 2048-iter Python loop.

### What was done
Three precompute steps added at the end of `G1PrimitiveSequenceDataset.__init__`:

1. **All motion → single GPU tensor**. `(N, T, D)` float32 (~960 MB train + ~340 MB val per rank) moved to `self.device` once. Replaces all per-call `torch.tensor(numpy)` + `.to(device)`.
2. **All unique texts pre-encoded via batched CLIP**. ~184 unique texts → `(184, 512)` table on device. Built alongside a `text_to_idx` map.
3. **Per-primitive text-index list**. Each dataset entry stores its text indices into the table; no more `_get_text_embedding` calls at runtime.

`_build_primitive_batch` rewritten:
- Takes a 1-D index array instead of a list of dicts.
- `motion_tensor = self.all_motion_tensor.index_select(0, indices_tensor)` — one GPU fancy-index op replaces 512 small H2D copies.
- Random text choice is a tiny Python loop (just `random.randrange` + list lookup, no torch).
- `text_embeddings = self.all_text_embeddings.index_select(0, ...)` — one GPU lookup replaces 512 dict accesses.

`get_batch` updated to pass numpy index arrays (1D for `num_primitive=1`, 2D `(num_primitive, B)` for >1) instead of building dict lists.

### Problems & Solutions
- **Problem [13:00]:** Asymmetric GPU utilization in DDP (cuda:0 73%, cuda:1 26%) — initially thought it was Blackwell PRO 6000 vs RTX 5090 heterogeneity, but user confirmed both cards are similar speed.
  - **Solution:** Real cause is `get_batch()` doing ~30 ms of CPU work per call. With `num_primitive=4`, that's ~120 ms per outer step pinned on CPU, leaving the GPU starving for data. The 26% rank was idle waiting at all-reduce because it was data-starved more often. Fix: precompute everything at init.
- **Problem [13:10]:** Heavy per-step CPU work breakdown.
  - **Solution:** Per `get_batch(512)` × `num_primitive=4` = 2048 inner iters, each calling `_data_to_tensor` (6 `torch.tensor` + `cat`), `.to(device)` (small H2D copy with PCIe latency), and `_get_text_embedding` (dict lookup). Replaced with on-device `index_select` lookups → ~30ms → ~0.5ms per call (~60×).

### Key findings
- Per-call CPU work in `get_batch`: ~30 ms → ~0.5 ms (~60× speedup). Across `num_primitive=4`, that's ~120 ms saved per outer step, plus removing per-step PCIe traffic.
- Memory cost: ~960 MB train + ~340 MB val per rank on GPU. Both 5090 (32 GB) and Blackwell PRO 6000 (96 GB) have plenty of headroom.
- Both `train_g1_mvae.py` and `train_g1_mld.py` use this dataset, so VAE training also benefits.
- DDP throughput at step 20 wasn't representative — included CUDA/cuDNN warmup + cold CLIP cache. Need to re-measure at steady state.
