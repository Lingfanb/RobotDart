## [13:33] Gap analysis: v6 vs original DART
**Summary:** User reported v6 still feels mediocre vs. original DART. Compared training configs and dataset distributions to identify the dominant cause.

### What was done
- Read original DART training command from [README.md:384](../README.md#L384).
- Compared with `mld_denoiser/g1_mld_v6/args.yaml`.
- Dumped text distribution from `data/mp_data_g1/Canonicalized_h2_f8_num1_fps30/train.pkl` (66,496 primitives, 2596 unique texts).

### Key findings

**Architecture and training are essentially identical to original DART**: 8-layer transformer, h_dim=512, latent=128, num_primitive=4, batch=1024, lr=1e-4, AMP, weight_*_delta=1e4.

**Real differences (ranked by likely impact):**

1. **Data scale (biggest)**: BABEL has ~9k+ sequences; we have 1612 train sequences. ~5–6× less data.
2. **Static-pose contamination**: top-5 texts in our G1 mp data are
   ```
   5332  stand
   3581  walk
   1669  tpose          ← T-pose, not motion
   1249  throw
   1122  transition to stand   ← static
   ```
   Static / quasi-static texts (stand + tpose + apose + transition to stand) total **8648 / 66496 = 13%**. Model is heavily biased toward "stand still."
3. **Training steps**: v6 used 80k+80k+80k = 240k. Original DART uses 100k+100k+100k = 300k. The rollout stages (2+3) are 40k shorter.
4. **Sampling scheme**: v6 uses `weight_scheme='text'` (pure inverse text frequency). Original DART uses `'text_samp:0.'` (mixed). Possibly over-aggressive on rare actions.
5. **Rare action coverage**: 1053 texts appear ≤5 times. "run forward" only has 12 examples vs "walk forward" 749.
6. **CLIP ViT-B/32** as text encoder (cos sim 0.85+ between motion prompts) — long-term issue, same as original DART.

### Concrete improvement options (proposed to user)
1. **Filter tpose/apose/transition** → regenerate mp_data. ~10–20% gain expected, ~30 min work.
2. **Try higher guidance scale** (7 or 10 instead of 5). 10 minutes to verify, no retrain.
3. **Extend training to 300k** (e.g., 80k + 100k + 120k stages). ~3.5h.
4. **Tune `weight_scheme`** — try `'text_samp:0.5'` to mix category-balanced and inverse-text sampling.
5. **Mirror augmentation** — flip left/right pose to double dataset. Need to write a `dof_6d` mirror function with G1 link index map.
6. **SEED dataset** (long-term, 22 GB G1 motion clips, license still pending HF approval).

### Next steps
- Wait for user to pick a path. Recommended: 1 + 2 first (fastest validation), then 5 (mirror) for permanent 2× data gain.

## [14:30] TextOp paper analysis (papers/TextOp.pdf)
**Summary:** Read TextOp (arXiv:2602.07439, 2026-02-07), an open-source DART→Unitree G1 adaptation with the **same goal** as ours. Their reported method achieves SOTA on BABEL validation. Identified 5 concrete improvements we should adopt.

### What was done
- Read all 20 pages of [papers/TextOp.pdf](../papers/TextOp.pdf) including main text, ablations, and appendix.
- Cross-checked their hyperparameters (Table VI), feature representation (Table XII), and ablation study (Table XIII) against our v6 config.
- Identified what's the same, what differs, and which differences matter.

### What is identical to v6 (architecture is fine)
- Robot: Unitree G1, 29 body DOFs (strip 14 hand DOFs from raw 43)
- Data source: AMASS + BABEL via GMR retarget
- Architecture: Transformer LDM, latent_dim=128, hidden=512, 8 layers, 4 heads
- Encoder: CLIP
- T_history=2, T_future=8, **N_primitive=4**
- CFG σ=5
- **Their Table XIII ablation confirms our exact hyperparams (N_prim=4, hidden=512, layers=8, σ_CFG=5) are all optimal.**

### Critical differences (ranked by ROI)

#### 1. ⭐⭐⭐ Feature representation: 69-dim vs our 360-dim (BIGGEST WIN)
TextOp Table XII directly compares representations and shows their 69-dim beats DART+Retarget, BeyondMimic, HumanML3D, RobotMDM on FID, R-precision, MM-Dist (Table III).

| Component                          | TextOp 69-dim | Our 360-dim |
|---|---|---|
| Root roll/pitch (trig encoding)    | 4 ✓           | ✗           |
| Root rotation 6D + diff 6D         | 12 ✓          | 6 (only diff) |
| Root yaw velocity                  | 1 ✓           | ✗           |
| Root z angular velocity            | 1 ✓           | ✗           |
| **DoF angle (1D × 29)**            | **29 ✓**      | ✗ (use 6D × 29 = 174!) |
| **DoF velocity (1D × 29)**         | **29 ✓**      | ✗           |
| Body link position (87)            | ✗             | 87 ✓        |
| Body link velocity (87)            | ✗             | 87 ✓        |
| **Foot contact (2)**               | **2 ✓**       | ✗           |
| transl(3) + transl_delta(3)        | ✗             | 6 ✓         |

Quote (page 8): *"the compact DoF-based motion representation reduces the inherent modeling complexity relative to HumanML3D-style representations"*.

Our 360-dim is the worst combo: 6× redundancy in DoF (174 vs 29), redundant link positions (87+87), and **no foot contact** → no ground constraint → walk slips/slow.

#### 2. ⭐⭐⭐ Data scale: 15× more raw motion
- TextOp: 40,767 raw GMR clips → 33,478 filtered → 83,478 BABEL segments
- Us: 2,660 raw GMR clips → 2,187 filtered → ~30,000 segments

They retarget the **full AMASS**; we only have a subset. Need to investigate why GMR_filtered/ has only 2660 (likely pipeline only ran on a partial AMASS subset).

#### 3. ⭐⭐ Diffusion steps: 5 vs our 10
Table XIII ablation shows steps=5 gives best FID/diversity, beating 7 and 10. Faster inference, no quality loss. Trivial to change.

#### 4. ⭐⭐ Loss weights are 5–6 orders of magnitude smaller
| λ            | TextOp | Ours    |
|---|---|---|
| simple       | 1.0    | 1.0 ✓   |
| KL           | 1e-4   | (VAE)   |
| **trans**    | **0.05** | **1e4** |
| **rot**      | **1e-2** | **1e4** |
| **dof**      | **0.03** | **1e4** |
| **dof_vel**  | **1e-5** | ✗       |
| **contact**  | **0.01** | ✗       |

Our 1e4 came from original DART on SMPL-X. TextOp re-tuned these for G1. Our model is essentially overfitting to delta predictions because the delta loss dominates by 1e6×.

#### 5. ⭐ Tracker data augmentation (relevant for Phase 5)
TextOp trains the tracking policy on (real motion + 5,368 LDM-generated clips, 31.48 hours). Table V shows `TextOp-M+G > TextOp-G > TextOp-M`. Important when we get to RL steering / tracking policy.

| Max rollout prob (LDM)      | 0.8    | 1.0?      |

### Key findings
- **Our architecture and most hyperparams are correct.** The Table XIII ablation in TextOp is essentially a vindication of the DART hyperparameter family.
- **Two things are wrong**: feature representation (we use 5× too many dims with the wrong content) and loss weights (5–6 OOM too large for the delta terms).
- **One thing is missing**: foot contact in features → no ground constraint signal.
- **One thing is structurally limited**: data scale 15× smaller. Retargeting full AMASS is prerequisite for closing the gap.
- TextOp open-sources code at `text-op.github.io` — should pull repo to get the exact 69-dim feature implementation and loss values rather than re-deriving.
